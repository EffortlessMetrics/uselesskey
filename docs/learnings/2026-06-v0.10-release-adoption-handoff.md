# Release Adoption Work Stops at the Source Boundary

## Trigger

The v0.10.0 release-adoption lane proved CLI, facade, schema, receipt, command
ledger, support-tier, and package surfaces in `uselesskey-swarm`. The actual
release still had to move through `EffortlessMetrics/uselesskey`, which owns
publication, tags, and release records.

## What Changed

The source repo consumed the swarm handoff through explicit source PRs instead
of polishing the swarm repo after the handoff boundary:

```text
uselesskey-swarm
  prepares, tests, audits, and hands off

EffortlessMetrics/uselesskey
  owns the candidate, publication, tag, and release record
```

The package-proven candidate SHA, not the later evidence-only `main` merge
commit, became the publish and tag authority.

## Evidence

- #862 imported the v0.10 adoption handoff.
- #864 prepared the v0.10.0 source candidate.
- #865 proved packaged adoption and produced candidate SHA
  `5ea65e1cc6309042731cf4ec91cb39f00a91253a`.
- #866 recorded immutable candidate evidence on source `main`.
- `v0.10.0` was published and tagged from
  `5ea65e1cc6309042731cf4ec91cb39f00a91253a`.
- Published-version CLI and facade smokes passed from the crates.io version.

## Rule to Keep

After a swarm handoff, do not keep polishing the swarm repo under the completed
lane. Move release-candidate, publication, tag, and post-release evidence work
into the source repository.

If a later source evidence commit changes package hashes, either publish from
the proven candidate SHA or regenerate the package evidence on the later commit.
